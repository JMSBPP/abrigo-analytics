DANE has therefore done the Rev.3.1 → Rev.4 mapping inside the Empalme publishing for 2010-2014.*"

That finding **was used** as the central justification for picking Option α: anti-fishing-clean because no Rev.3.1 → Rev.4 author crosswalk is required anywhere in the 2010-2026 window.

## §2. The pathology — empalme files for 2010-2014 carry Rev.3 codes, not Rev.4

Re-inspection of the actual published Empalme files (sampling the January Ocupados.CSV from each year) yields the following column-presence table:

*The 2020 empalme catalog (`GEIH_EMPALME_2020_ACT.zip`) drops the legacy RAMA4D column entirely and ships RAMA4D_R4 as the canonical sector code.

The 2010 and 2014 RAMA4D codes (5211 = "Comercio al por menor de productos de uso doméstico", 9500 = "Hogares privados con servicio doméstico", 6031 = "Transporte por carretera", 8511 = "Actividades de hospitales", 1810 = "Fabricación de prendas de vestir", 8043 = "Educación universitaria") are **classic CIIU Rev. 3 a.c. codes**. The Rev.4 equivalents (4711, 9700, 4923, 8610, 1410, 8530) appear only from 2015 onward in the published RAMA4D_R4 column.

## §3. Why this contradicts the Option-α pin

Plan v2.2 Task 1.1 Step 0 (CORRECTIONS-α) pinned the following harmonization rule for the 2010-2020 era:

> | 2010-01 → 2020-12 (132 mo) | GEIH Empalme catalogs cid 755-765 (per-year ZIPs) | Comma-separated, UTF-8, single-file (no Cabecera/Resto split); FEX_C pre-incorporates empalme factor; **sector codes already in CIIU Rev.4 a.c.** | `FEX_C` | `RAMA4D` |

The bolded clause is data-falsified for **2010, 2011, 2012, 2013, 2014** — those 60 months use Rev.3 codes in the RAMA4D column. Reading those codes against the spec's `services_set = CIIU Rev.4 a.c. sections {G, H, I, J, K, L, M, N, O, P, Q, R, S, T}` would produce systematically misclassified Y values (e.g., Rev.3 5211 lives in section G under Rev.3's G-section definition AND under Rev.4's G-section definition, but Rev.3 9500 lives in section P-section "Households as Employers" under Rev.3 vs. T-section under Rev.4 — section-letter mappings are NOT 1-to-1 across revisions).

For 2015-2019 (60 months), the empalme file ships BOTH RAMA4D (Rev.3) and RAMA4D_R4 (Rev.4) — DANE pre-applied the crosswalk and the user can read the Rev.4 column directly. This sub-window is anti-fishing-clean.

For 2020 (12 months), the empalme file ships only RAMA4D_R4 (Rev.4). Anti-fishing-clean.

So the empalme series breaks down into:
- **2010-01 → 2014-12 (60 months): Rev.3 only — requires external crosswalk** (the precise pathology Option α was supposed to eliminate)
- **2015-01 → 2020-12 (72 months): Rev.4 available via RAMA4D_R4** (anti-fishing-clean)

For 2021 (Marco-2018 semester archives) and 2022-2026 (Marco-2018 native): RAMA4D_R4 is the canonical column; anti-fishing-clean.

## §4. Why the Step-0 disposition memo's verification claim was wrong

The Step-0 memo's "verified" assertion was based on inspecting only the column **header** of the 2010 + 2014 Empalme Ocupados.CSV files (the column name "RAMA4D" was correctly noted). The column **values** were not sampled; the disposition memo author appears to have assumed RAMA4D-column-name implies Rev.4-coding because the disposition's own §2.1 had asserted "sectoral codes harmonized to CIIU Rev.4 a.c. directly." The actual code values were not cross-checked against a Rev.4 codebook.

This is the precise failure mode `feedback_real_data_over_mocks` and `feedback_pathological_halt_anti_fishing_checkpoint` are designed to surface: schema-name verification is necessary but not sufficient; value-content verification must precede any harmonization-rule pin. The Step-0 memo's verification was incomplete.
