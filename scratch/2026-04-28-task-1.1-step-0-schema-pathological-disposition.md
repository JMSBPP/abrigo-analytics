## §2. Era-boundary inventory — verified DANE-resolvable break-points

Three break-points exist in the 2008-2026 window. Two have deterministic DANE-published harmonization rules; one does not.

### §2.1 Marco-2005 → Marco-2018 master-sample-frame change (DANE-resolvable for 2010-2020 only)

- DANE published the **GEIH Empalme series**: per-month re-published microdata with the empalme factor pre-applied to the expansion weight, pre-aggregated to national level (no Cabecera/Resto split), with sectoral codes harmonized to CIIU Rev.4 a.c. directly. Schema verified: columns `DIRECTORIO`, `SECUENCIA_P`, `ORDEN`, `OCI`, `FEX_C`, `RAMA4D`, `RAMA2D`, `P6040`, `AREA`, `CLASE`, `DPTO`, `ANO`, `MES`, `PERIODO`. One ZIP per year (cid 755-765), ~210-260 MB each, containing 12 inner monthly ZIPs.
- The DANE empalme nota técnica explicitly states (§3.3 verbatim from PDF): *"se construye un factor por cada mes a partir de **enero de 2010 hasta diciembre 2020** que secuencialmente ajusta una parte del nivel de la serie de ocupados. … se divide a θ en **132 partes** (son los meses desde enero 2010 hasta diciembre de 2020)"*. The empalme is **explicitly defined for the 2010-01 to 2020-12 window only** — 132 months. Pre-2010 data is OUT OF SCOPE for the published empalme.
- For 2010-01 through 2020-12 (132 months): **DANE-published harmonization rule available**; harmonization rule = "use the GEIH Empalme zip files (cid 755-765) directly; the FEX_C column already incorporates the empalme factor and the sector codes are already in Rev.4 a.c."
- For 2021-01 through 2021-12 (12 months, parallel-collection year): **DANE-published harmonization available** via the `GEIH - Marco-2018(I.Semestre).zip` and `GEIH_Marco_2018(II. semestre).zip` archives within cid 701. Schema verified: matches the 2022+ Marco-2018 native schema (RAMA4D_R4 + FEX_C18 columns, single-file-not-Cabecera/Resto). Harmonization rule = "for 2021, prefer the Marco-2018 semester archives over the per-month native files."
- For 2022-01 through 2026-03 (51 months as of 2026-03): native Marco-2018 publishing pre-incorporates everything; no harmonization needed. RAMA4D_R4 + FEX_C18 schema is stable across 2022, 2023, 2024, 2025, 2026.
- **For 2008-01 through 2009-12 (24 months): NO DANE-published Marco-2005 → Marco-2018 empalme exists. The 132-month empalme expressly excludes pre-2010 data.**

### §2.2 CIIU Rev. 3.1 → CIIU Rev. 4 a.c. classification change (Resolution 066 of 2012-01-31)

- 2008-01 through 2011-12: GEIH native Ocupados schema uses Rev. 3.1 sector codes in the column `RAMA4D` (4-digit) and `RAMA2D` (2-digit). Verified in raw 2008-01 + 2012-01 native CSVs (the column "RAMA4D" exists in both eras, but the *coding* changes from Rev.3.1 to Rev.4 a.c. in 2012-01 per DANE Resolution 066).
- 2012-01 onward (native): RAMA4D + RAMA2D contain Rev. 4 a.c. codes directly. Schema-rename to `RAMA4D_R4` + `RAMA2D_R4` arrives in 2022+ Marco-2018 publishing.
- 2015-2019 supplemental: DANE published the CIIU-4 a.c. crosswalk catalog (cid 661) that re-classifies 2015-2019 Ocupados rows from Rev.3.1/Rev.3 a.c. into Rev.4 a.c. via a person-level per-DIRECTORIO/SECUENCIA_P/ORDEN join. Schema verified: 5 files (one per year 2015-2019), columns `DIRECTORIO`, `SECUENCIA_P`, `ORDEN`, `RAMA4D_R4`, `RAMA2D_R4`, `RAMA4D_D_R4`, `RAMA2D_D_R4`. **This catalog covers 2015-2019, NOT 2008-2014.**
- For 2010-2014 in the GEIH Empalme series: sector codes are **already in Rev.4 a.c.** in the Empalme files (verified — the 2010-01 and 2014-01 Empalme Ocupados.CSV contain RAMA4D + RAMA2D with Rev.4 a.c. codes, NOT Rev.3.1). DANE has therefore **done the Rev.3.1 → Rev.4 mapping inside the Empalme publishing for 2010-2014.**
- For **2008-2009 native files**: sector codes are CIIU Rev.3.1 in the `RAMA4D` column. To map to the spec-required services_set = CIIU Rev.4 a.c. sections {G, H, I, J, K, L, M, N, O, P, Q, R, S, T}, an external Rev.3.1 → Rev.4 correspondence table is required. **DANE publishes a printed correspondence table (PDF document `https://www.dane.gov.co/files/sen/nomenclatura/ciiu/CIIU_Rev_4_AC2022.pdf` and the CIIU-Rev-4 disposed online tool at `https://clasificaciones.dane.gov.co/ciiu4-0/ciiu4_dispone`)** but at the per-row 4-digit code level the mapping is in some cells **non-deterministic** (one Rev.3.1 4-digit code maps to multiple Rev.4 4-digit codes depending on the activity description), which forces author-judgment-driven case-by-case classification — exactly the discretion that spec §9 anti-fishing-bans.

### §2.3 Per-month CSV column-rename / encoding / separator drift (DANE-resolvable)

These are minor and deterministic:
- 2008-2009 CSVs: comma-separated, UTF-8 (with BOM), 24 files per month split by Cabecera/Resto/Área with `(1)` filename suffix
- 2010-2011 native: comma-separated, UTF-8, similar Cabecera/Resto/Área split
- 2012-2021 native (per-month, Marco-2005): semicolon-separated, UTF-8, Cabecera/Resto/Área split (some 2021 files use Latin-1 encoding for special characters)
- 2010-2020 Empalme: comma-separated, UTF-8, single-file (no Cabecera/Resto split), AREA + CLASE columns differentiate
- 2022-2025 Marco-2018 native: comma-separated, **Latin-1 encoding** (mojibake risk on Spanish accents in column headers if read as UTF-8), single-file
- 2026 native: semicolon-separated, UTF-8, single-file, BUT with new prefix columns `PERIODO`, `MES`, `PER` ahead of `DIRECTORIO`

All of these are pin-able with a per-era column-rename + encoding-detect rule. **Deterministic harmonization rule**: union of the per-era schemas; primary keys `DIRECTORIO`, `SECUENCIA_P`, `ORDEN` are stable across all eras; sector field name maps {RAMA4D, RAMA4D_R4} → unified `_rama4d_rev4`; FEX field name maps {fex_c_2011, FEX_C, FEX_C18} → unified `_fex_c`; age field `P6040` is stable.

## §3. The pathology — 2008-2009 is the single non-pinnable break-point

The 2008-01 through 2009-12 window (24 months out of the spec-pinned 218-month window) requires **TWO simultaneous harmonization actions** that DANE did NOT publish for that pre-empalme era:

1. **Marco-2005 → Marco-2018 empalme** — explicitly out of scope per the DANE nota técnica (§3.3 quote above)
2. **CIIU Rev.3.1 → CIIU Rev.4 a.c. crosswalk** — for which DANE publishes only a (partially) ambiguous per-4-digit-code correspondence PDF that requires author judgment in 1-to-many cases

For 2010-2020, DANE published the GEIH Empalme series that does BOTH simultaneously (and was published expressly for time-series analyses spanning 2021). For 2021, DANE published the Marco-2018 semester ZIPs. For 2022+, native Marco-2018 publishes everything pre-applied. **2008-2009 is the singular orphan window.**

Per plan Task 1.1 Step 0 (NEW per SPM D5 BLOCK): *"If any break-point cannot be pre-pinned with a deterministic harmonization rule, HALT per the anti-fishing protocol below"*. This break-point fits exactly that condition.

Per spec §9.5: HALT-disposition path. Per spec §9: pinning a fishy / author-judgment harmonization rule (e.g., choosing one branch of a 1-to-many Rev.3.1 → Rev.4 crosswalk to make the 2008-2009 share match smoothly) **would itself be an anti-fishing violation** — Y construction discretion.

## §4. Pivot options (≥3 enumerated; user picks)

### Option α: shorten the pinned sample window to 2010-01 → 2026-03 inclusive

- New sample window: 2010-01 → 2026-03 = **195 monthly observations**
- After lag-12 loss: **183 candidate monthly Y observations**
- Far above N_MIN_OBS = 75 (Phase-A.0 floor).
- Implementation simplicity: PRIMARY harmonization is "for 2010-2020, use Empalme catalogs (cid 755-765); for 2021, use Marco-2018 semester archives in cid 701; for 2022-2026, use Marco-2018 native".
- **Deterministic, DANE-canonical, anti-fishing-safe.**
- Spec §9.3 sample-window-immutability invariant requires a CORRECTIONS block + 3-way review of the CORRECTIONS revision before this lands. **The pivot itself is data-availability-driven, not data-driven**, which is the legal channel for §9.3 amendment.
- Cost: 2 calendar years of pre-empalme monthly data dropped (2008-01 → 2009-12 = 24 months). The user's lag X regressor is COP/USD lagged 6-12 months; 2008-2010 is also where the Colombia commodity-boom peaks live, so this drops a notable macro-volatility window from the OLS sample. However, the same 2010-onward window is 100% within the empalme-supported era and is the standard DANE recommendation for Marco-2018-comparable time-series analysis.

### Option β: keep 2008-2026 window; for 2008-2009, apply DANE Rev.3.1 → Rev.4 crosswalk + KEEP RAW Marco-2005 expansion factors (no empalme for 2008-2009)

- For 2008-2009: apply the DANE-published CIIU Rev.3.1 → Rev.4 a.c. correspondence at the section-letter level (G-T, A, B, C, D, E, F) — the broad-services aggregation hides most of the 1-to-many per-4-digit cases at the section-letter granularity (a Rev.3.1 4-digit "wholesale food" code stays in section G of Rev.4; a Rev.3.1 "hotel" stays in I; etc.). The author-judgment-required cases at the 4-digit level mostly resolve at the section-letter level. **Section-letter resolution must be verified for every Rev.3.1 4-digit code in the 2008-2009 universe before this option is operational.**
- For 2008-2009 expansion factors: use the raw Marco-2005-frame `fex_c_2011` directly (no empalme factor available). This means the 2008-2009 raw share `Y_t` is denominated in Marco-2005-population units while 2010-2020 is in Marco-2018-population units. Spec §6 acknowledges this exact concern via R1 (2021 regime dummy) — Option β extends that logic by treating 2008-2009 as a methodologically distinct sub-regime.
- For 2010-2026: same as Option α primary.
- Cost: introduces a methodologically inhomogeneous 2-year sub-period that survives PRIMARY estimation only via implicit "trust the Y series across this break" — which is exactly what spec §6 was designed to NOT do for the 2021 break (where DANE did publish empalme; the spec rejects per-author-judgment Marco-2005-vs-Marco-2018 patches when DANE publishes empalme). Symmetry argues Option β SHOULD be rejected for the 2008-2009 patch where DANE explicitly did NOT publish empalme. **Author-judgment Y-construction across the 2008-2009 break is the precise anti-fishing concern §9.2 immutability is designed to prevent.**
- Operational risk: the section-letter Rev.3.1 → Rev.4 mapping must be verified case-by-case for every 4-digit Rev.3.1 code that appears in 2008-2009 GEIH Ocupados rows. Even if 100% of cases resolve cleanly at the section-letter level, the verification work is substantive.
- Robustness diagnostic: this option lets us add a "2008-2009 dummy" to R1 (alongside the 2021 dummy) — but adds a second methodological-regime indicator, complicating the lag-X composite-β interpretation.

### Option γ: keep 2008-2026 window; for 2008-2009, apply Rev.3.1 → Rev.4 crosswalk AND a manually-constructed empalme factor (extending DANE's 132-month methodology backward 24 months to 2008)

- For 2008-2009: apply the DANE Rev.3.1 → Rev.4 correspondence (same caveat as Option β); additionally, construct the empalme factor for months 2008-01 through 2009-12 by extending DANE's stepwise formula `Ponderador_i = 1 - (i × θ / 132)` backward 24 months (i.e., for 2008-01, the factor is `1 - ((-24) × θ / 132)` = `1 + (24 × θ / 132)`).
- Cost: this is **author-constructed**, NOT DANE-published. The 132-month split is explicitly grounded by DANE in the parallel-collection 2021 endpoint and the assumed "transition simulation" interpretation of the marco-2005 → marco-2018 shift; extending it backward two more years adds 24 months of unobserved-counterfactual extrapolation that DANE did not endorse. **This option is a non-DANE-canonical patch and would not be defensible at 3-way review.**
- **Operational viability: LOW.** Treat this option as a non-starter unless the user wants to pursue it as a pre-registered Robustness check (R5+) and not as the primary harmonization. Since spec §7 R1-R4 are already pinned and §9.2 fixes the robustness universe, adding R5 requires a CORRECTIONS block + 3-way review.

### Option δ: shorten further to 2012-01 → 2026-03 (CIIU-Rev.4 native era)

- Sample window: 2012-01 → 2026-03 = **171 monthly observations**; after lag-12 loss: 159 candidate Y observations.
- For 2012-2020: still uses Empalme catalogs (the empalme starts 2010-01; 2012-01 onwards is fully covered). Simplification gain: NO Rev.3.1 → Rev.4 crosswalk required *anywhere* in the window because every native file from 2012-01 onward already uses Rev.4 a.c. (per Resolution 066 of 2012-01-31).
- Cost: drops 4 additional years (2008-2011 = 48 months) on top of Option α's 24 months. Total monthly drop = 48 months vs. Option α.
- Trade-off: maximal harmonization simplicity at the cost of further sample shrinkage. Still well above N_MIN=75. The post-2010-commodity-boom window may underweight the most-relevant FX-X transmission events.

### Option ε: shorten to 2010-01 → 2020-12 (empalme-only window; pre-empalme AND post-empalme excluded)

- Sample window: 2010-01 → 2020-12 = **132 monthly observations**; after lag-12 loss: 120.
- Excludes both 2008-2009 AND 2021-2026.
- Operational simplicity: ONLY the Empalme catalogs are needed. No Marco-2018 semester archives, no 2022+ native files.
- Cost: drops 2021-2026 = 63 additional months (vs. Option α). The post-2020 era is methodologically the cleanest (Marco-2018-only); excluding it is anti-fishing-paradoxical.
- **NOT recommended.** Listed for completeness only; the post-2020 data is more reliable than pre-2010, so excluding the post and keeping the pre is upside-down.
